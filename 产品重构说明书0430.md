## 2. 背景说明

pm-mem 原始定位是基于 ReMem（Self-Evolving Memory for LLM Agents）方法论实现的自演化记忆管理系统，核心能力包括：

- 检索记忆。
- 选择 Think / Refine / Act 动作。
- 通过 Refine 修改记忆。
- 通过 Act 输出结果并写入新记忆。
- 将记忆持久化，形成长期演化能力。

近期项目新增了短剧创作、分镜、外部作品导入、六层 Markdown 分层记忆和 Web 管理页等能力。短剧业务天然需要长期记忆管理，包括故事设定、人物背景、剧情进展、角色性格、分季分集内容和分镜档案等信息的一致性维护。

当前问题在于：新增短剧工作流和外部作品导入流程分别形成了独立的记忆写入链路，未统一经过原 ReMemAgent 的记忆演化流程。长期看，这会导致记忆体系分裂、演化轨迹不统一、冲突检测能力不足、核心方法论无法持续复用。

## 3. 问题定义

### 3.1 新增短剧工作流绕过 ReMemAgent

当前短剧工作流主要通过读取六层 Markdown 内容构造 prompt，调用 LLM 后再将输出解析并写回 Markdown。该流程具备业务可用性，但没有经过 ReMem 的完整闭环：

```text
Retrieve -> Think -> Refine -> Act -> Memory Append -> Persist
```

因此，短剧工作流中的记忆更新更接近“业务写文件”，而不是“记忆自演化”。

### 3.2 外部作品导入形成另一套分层导入体系

当前外部导入流程由导入协调器、分层导入智能体、本地代理模型和 DeepSeek 备用通道组成。该流程可以生成六层 Markdown 记忆，但核心行为是“导入草稿并写入目标层”，而不是通过 ReMem 进行已有记忆检索、冲突分析、合并、精炼和演化。

重复导入、内容冲突、旧记忆保留、新记忆合并等场景，应由统一的记忆演化机制处理。

### 3.3 现有架构扩展点不足

原 ReMemAgent 与 MemoryBank / JSON Persistence 的耦合较强，难以直接承载 Markdown 分层记忆、角色 prompt、锁定层、分季分集、外部导入元数据等业务场景。

但重构方向不应是新增一个业务包装类，例如 `LayeredReMemAgent`。该方式会在核心能力旁边再形成一层业务代理，不符合“干净彻底重构”的目标。

## 4. 重构目标

### 4.1 产品目标

将 pm-mem 重构为：

```text
支持角色注入和存储后端注入的领域无关 ReMem 自演化记忆引擎。
```

短剧分层记忆是该引擎的一个业务落地形态，而不是独立于 ReMem 的另一套记忆系统。

### 4.2 技术目标

1. 直接重构 `ReMemAgent`，使其成为唯一的记忆演化入口。
2. 保持 `ReMemAgent` 与短剧业务无关。
3. 支持注入不同角色实例，例如编剧、制片人、分镜师、一致性校验员。
4. 支持注入不同记忆存储后端，例如 JSON MemoryBank、Markdown 六层记忆。
5. 所有自动记忆写入都必须由 ReMem 的 Refine 或 Act 后写入完成。
6. 短剧创作、分镜生成、一致性校验、外部作品导入统一进入 `ReMemAgent.run_task()`。
7. Web 管理页能够展示记忆演化过程，包括检索、思考、编辑、写入和冲突记录。

### 4.3 非目标

1. 不新增 `LayeredReMemAgent` 等业务包装 Agent。
2. 不在 `ReMemAgent` 中硬编码短剧字段、层名称、角色名称或作品目录。
3. 不取消现有 JSON MemoryBank 能力。
4. 不取消 Markdown 六层记忆的人类可读、可审核、可手工编辑能力。

## 5. 核心设计原则

### 5.1 ReMemAgent 是唯一记忆演化入口

所有需要自动改变记忆的业务动作都必须进入 `ReMemAgent`：

- 普通任务执行。
- 短剧剧本生成。
- 分镜生成。
- 一致性校验。
- 外部作品导入。
- 重复导入后的记忆合并。

### 5.2 ReMemAgent 保持领域无关

`ReMemAgent` 只关心通用流程：

```text
记忆检索 -> 动作选择 -> Think -> Refine -> Act -> 记忆写入 -> 持久化 -> 轨迹记录
```

它不关心：

- 是否是短剧。
- 是否有六层 Markdown。
- 是否是编剧或分镜师。
- 是否是第几季第几集。
- 是否来自外部作品导入。

这些信息通过角色实例、上下文元数据和存储后端表达。

### 5.3 角色负责专业判断

角色实例不负责直接写文件。角色负责提供：

- 角色 prompt。
- 检索偏好。
- Think 阶段关注点。
- Refine 阶段记忆提取规则。
- Act 阶段输出规范。
- 可执行和禁止执行的记忆操作约束。

### 5.4 MemoryStore 负责记忆载体

存储后端负责：

- 如何读取记忆。
- 如何检索记忆。
- 如何应用记忆操作。
- 如何处理锁定、冲突、版本和持久化。

`ReMemAgent` 不直接依赖 Markdown 文件或 JSON 文件。

## 7. 核心抽象

### 7.1 TaskContext

`TaskContext` 用于携带任务元数据。该对象保持领域无关，业务字段统一放入 `metadata`。

```python
class TaskContext:
    task_type: str
    source: str
    role_id: str
    metadata: dict
```

短剧示例：

```json
{
  "task_type": "script_generation",
  "source": "short_drama_workflow",
  "role_id": "screenwriter",
  "metadata": {
    "work_id": "uuid",
    "season": 1,
    "episode": 3,
    "target_layers": [
      "core_setting",
      "character_profile",
      "plot_context",
      "script_archive"
    ]
  }
}
```

### 7.2 MemoryRecord

`MemoryRecord` 是 ReMemAgent 看到的统一记忆单元。

```python
class MemoryRecord:
    id: str
    content: str
    metadata: dict
    score: float | None
```

JSON MemoryBank 可以将 `MemoryEntry` 转换为 `MemoryRecord`。Markdown 六层记忆可以将层内容、章节、段落或结构化块转换为 `MemoryRecord`。

### 7.3 MemoryOperation

`MemoryOperation` 是 Refine 阶段输出的统一记忆操作。

```python
class MemoryOperation:
    operation_type: str
    target: str
    content: str
    metadata: dict
```

通用操作类型：

- `add`
- `append`
- `replace`
- `delete`
- `merge`
- `relabel`
- `flag_conflict`
- `no_op`

短剧层写入不应引入 ReMemAgent 特有分支，而应通过 `target` 和 `metadata` 表达：

```json
{
  "operation_type": "append",
  "target": "memory.layer",
  "content": "第3集新增剧情事实...",
  "metadata": {
    "work_id": "uuid",
    "layer_id": "plot_context",
    "season": 1,
    "episode": 3
  }
}
```

### 7.4 MemoryRole

`MemoryRole` 是角色注入接口。

```python
class MemoryRole:
    role_id: str
    role_name: str

    def system_prompt(self) -> str: ...

    def retrieval_policy(self, context: TaskContext) -> dict: ...

    def think_instructions(self, context: TaskContext) -> str: ...

    def refine_instructions(self, context: TaskContext) -> str: ...

    def act_instructions(self, context: TaskContext) -> str: ...

    def allowed_operations(self, context: TaskContext) -> list[str]: ...

    def memory_update_schema(self, context: TaskContext) -> dict: ...
```

角色实例通过 prompt 和策略影响 ReMemAgent 的执行质量，但不改变 ReMemAgent 的领域无关性。

### 7.5 MemoryStore

`MemoryStore` 是记忆存储接口。

```python
class MemoryStore:
    def retrieve(
        self,
        query: str,
        context: TaskContext,
        role: MemoryRole,
        k: int
    ) -> list[MemoryRecord]: ...

    def apply_operations(
        self,
        operations: list[MemoryOperation],
        context: TaskContext,
        role: MemoryRole
    ) -> dict: ...

    def append_task_result(
        self,
        task_input: str,
        action_output: str,
        feedback: str,
        context: TaskContext,
        role: MemoryRole
    ) -> MemoryRecord | None: ...

    def save(self) -> bool: ...
```

### 7.6 TraceStore

`TraceStore` 负责记录演化轨迹。

```python
class TraceStore:
    def record_task_started(...): ...
    def record_retrieval(...): ...
    def record_think(...): ...
    def record_refine(...): ...
    def record_act(...): ...
    def record_task_finished(...): ...
```

轨迹应至少包含：

- 任务输入。
- 角色信息。
- 检索到的记忆。
- Think 输出。
- Refine 原始输出。
- 解析后的 MemoryOperation。
- 实际写入结果。
- 冲突和跳过原因。
- Act 最终输出。

## 8. ReMemAgent 改造方案

### 8.1 新初始化方式

```python
agent = ReMemAgent(
    llm=llm_client,
    memory_store=memory_store,
    trace_store=trace_store,
    max_iterations=8,
    retrieval_k=5,
)
```

兼容旧用法时，可默认构造 `JsonMemoryStore`：

```python
agent = ReMemAgent(llm=llm_client)
```

### 8.2 新任务执行接口

```python
result = agent.run_task(
    task_input="写第3集开场，突出女主反击动机。",
    role=ScreenwriterRole(),
    context=TaskContext(
        task_type="script_generation",
        source="short_drama_workflow",
        role_id="screenwriter",
        metadata={
            "work_id": "...",
            "season": 1,
            "episode": 3
        }
    )
)
```

### 8.3 主流程

```text
1. 接收 task_input、role、context
2. 调用 memory_store.retrieve()
3. 构建动作选择 prompt
4. LLM 选择 Think / Refine / Act
5. Think 阶段：结合 role prompt 和检索记忆生成推理轨迹
6. Refine 阶段：生成 MemoryOperation 列表
7. memory_store.apply_operations()
8. Act 阶段：生成最终输出
9. memory_store.append_task_result()
10. memory_store.save()
11. trace_store 记录完整演化轨迹
```

### 8.4 角色对各阶段的影响

| 阶段 | ReMemAgent 职责 | MemoryRole 职责 |
| --- | --- | --- |
| Retrieve | 发起检索请求 | 提供检索偏好 |
| Think | 构建并执行 Think prompt | 提供专业分析角度 |
| Refine | 解析并应用记忆操作 | 提供记忆更新规则 |
| Act | 生成最终输出 | 提供输出格式和质量标准 |
| Persist | 触发保存 | 不直接保存 |

## 9. 短剧角色设计

### 9.1 制片人角色

职责：

- 维护核心设定、题材定位、风格边界、商业方向。
- 判断哪些信息属于不可随意改写的核心记忆。
- 对锁定层冲突进行审核建议。

重点关注：

- 核心冲突。
- 世界观和背景。
- 人物关系的大方向。
- 已锁定设定。

### 9.2 编剧角色

职责：

- 维护人物档案、剧情推进、对白风格、分集内容。
- 从生成剧本中提取新增人物事实和情节事实。
- 保障角色性格和前后剧情一致。

重点关注：

- 人物动机。
- 人物性格。
- 情节因果。
- 分季分集连续性。
- 对白口吻。

### 9.3 分镜师角色

职责：

- 维护分镜档案、视觉连续性、场景调度和镜头信息。
- 从剧本或外部分镜中提取可复用视觉记忆。

重点关注：

- 场景。
- 景别。
- 镜头顺序。
- 人物站位。
- 道具和视觉风格。

### 9.4 一致性校验员角色

职责：

- 检查新内容与已有记忆的冲突。
- 对锁定层冲突生成待确认记录。
- 不直接覆盖核心设定。

重点关注：

- 设定冲突。
- 人物行为冲突。
- 时间线冲突。
- 剧情因果冲突。
- 分镜与剧本不一致。

## 10. Markdown 分层记忆接入方案

### 10.1 MarkdownLayerMemoryStore 职责

`MarkdownLayerMemoryStore` 是 `MemoryStore` 的实现之一，负责将六层 Markdown 作为 ReMem 的持久化记忆后端。

职责包括：

1. 根据 `work_id` 读取作品目录。
2. 将六层 Markdown 内容转换为 `MemoryRecord`。
3. 根据角色检索偏好调整候选记忆。
4. 将 `MemoryOperation` 写回对应 Markdown 层。
5. 处理锁定层保护。
6. 记录冲突、待确认变更和跳过原因。

### 10.2 六层记忆映射

| 层 | 业务含义 | 默认更新角色 |
| --- | --- | --- |
| work_metadata | 作品来源、导入状态、版本信息 | 制片人 |
| core_setting | 核心设定、世界观、不可改写规则 | 制片人 |
| character_profile | 人物背景、性格、关系、口吻 | 编剧 |
| plot_context | 分季分集剧情、时间线、因果关系 | 编剧 |
| script_archive | 剧本正文和分场档案 | 编剧 |
| storyboard_archive | 分镜正文和视觉档案 | 分镜师 |

### 10.3 锁定层规则

1. 锁定层不允许普通角色自动覆盖。
2. 当 Refine 需要修改锁定层时，MemoryStore 将操作转为 `flag_conflict`。
3. `flag_conflict` 写入待确认区域或演化日志。
4. 制片人或用户手动确认后，才允许修改锁定层内容。

## 11. 短剧工作流改造方案

### 11.1 改造前

```text
ShortDramaWorkflow
  -> get_layer_content_for_prompt()
  -> llm(prompt)
  -> update_memory_from_llm_output()
  -> Markdown
```

### 11.2 改造后

```text
ShortDramaWorkflow
  -> 构造 TaskContext
  -> 选择 MemoryRole
  -> ReMemAgent.run_task()
  -> MarkdownLayerMemoryStore 写入记忆
  -> TraceStore 记录演化轨迹
```

### 11.3 方法映射

| 当前方法 | 新角色 | task_type |
| --- | --- | --- |
| create_script_episode | ScreenwriterRole | script_generation |
| create_storyboard | StoryboardRole | storyboard_generation |
| consistency_check | ConsistencyReviewerRole | consistency_check |
| run_step | 由调用方传入 | generic_workflow_step |

### 11.4 强约束

`workflow.py` 不再直接调用 `update_memory_from_llm_output()` 进行最终业务记忆写入。所有自动写入必须由 `ReMemAgent` 产生的 `MemoryOperation` 或 Act 后写入完成。

## 12. 外部作品导入改造方案

### 12.1 改造前

```text
ExternalWorkImportCoordinator
  -> 分层导入 Agent 生成草稿
  -> 调用本地代理 LLM 或 DeepSeek
  -> 直接写入六层 Markdown
```

### 12.2 改造后

```text
ExternalWorkImportCoordinator
  -> 标准化外部 payload
  -> 构造 import TaskContext
  -> 按目标层选择 MemoryRole
  -> ReMemAgent.run_task()
  -> ReMem 检索已有记忆
  -> ReMem Refine 合并、补充、标记冲突
  -> ReMem Act 输出导入报告
  -> MarkdownLayerMemoryStore 写入结果
```

### 12.3 导入角色编排

| 导入内容 | 角色 | 目标层 |
| --- | --- | --- |
| 作品来源、外部 ID、URL | 制片人 | work_metadata |
| 核心设定、题材、风格 | 制片人 | core_setting |
| 人物信息 | 编剧 | character_profile |
| 剧情脉络 | 编剧 | plot_context |
| 剧本文本 | 编剧 | script_archive |
| 分镜文本和图片线索 | 分镜师 | storyboard_archive |
| 冲突和完整性检查 | 一致性校验员 | trace / conflict |

### 12.4 重复导入规则

重复导入同一 `source_system + external_work_id` 时：

1. 不创建重复作品。
2. 检索已有六层记忆。
3. 对比新旧内容。
4. 新事实进入 append 或 merge。
5. 冲突事实进入 flag_conflict。
6. 不允许简单覆盖已审核或锁定记忆。

## 15. 阶段计划

### 阶段一：核心抽象重构

目标：让 `ReMemAgent` 从固定 MemoryBank 实现升级为通用记忆演化内核。

交付物：

- `TaskContext`
- `MemoryRecord`
- `MemoryOperation`
- `MemoryRole`
- `MemoryStore`
- `TraceStore`
- `JsonMemoryStore`
- 默认 `GenericRole`
- 改造后的 `ReMemAgent`

验收标准：

1. 原 ReMem 单元测试和集成测试继续通过。
2. 旧 CLI 默认行为不退化。
3. 不传角色时使用 GenericRole。
4. 不传 MemoryStore 时兼容现有 JSON MemoryBank。

### 阶段二：短剧角色体系接入

目标：通过角色注入提升 Think、Refine、Act 的专业性。

交付物：

- `ProducerRole`
- `ScreenwriterRole`
- `StoryboardRole`
- `ConsistencyReviewerRole`
- 从 `roles/*.md` 加载角色 prompt 的 RoleFactory

验收标准：

1. 编剧角色关注人物、剧情、对白和分集连续性。
2. 分镜师角色关注镜头、场景、视觉连续性。
3. 制片人角色关注核心设定和锁定层。
4. 一致性校验员角色能够输出冲突报告。

## 18. 风险与控制

| 风险 | 影响 | 控制措施 |
| --- | --- | --- |
| ReMemAgent 重构影响旧功能 | 高 | 先实现 JsonMemoryStore 兼容层，保持旧测试通过 |
| 角色 prompt 过长 | 中 | 通过 MemoryRole 提供裁剪策略 |
| Markdown 自动写入破坏格式 | 高 | MemoryStore 统一负责 front matter 和正文写回 |
| LLM 输出操作不可解析 | 中 | 引入结构化 MemoryOperation schema 和回退 no_op |
| 锁定层被误改 | 高 | MemoryStore 层强制保护，ReMemAgent 无法绕过 |
| 外部导入重复覆盖旧记忆 | 高 | 重复导入必须走检索、合并、冲突标记 |
| Web 展示复杂度升高 | 中 | 先展示摘要，再支持展开详情 |

## 19. 验收总标准

1. `ReMemAgent` 是唯一自动记忆演化入口。
2. `ReMemAgent` 源码中不出现短剧层名、角色名、作品目录等业务硬编码。
3. 短剧剧本生成通过注入编剧角色进入 ReMemAgent。
4. 分镜生成通过注入分镜师角色进入 ReMemAgent。
5. 一致性校验通过注入一致性校验员角色进入 ReMemAgent。
6. 外部作品导入通过 ReMemAgent 完成记忆合并和冲突检测。
7. 六层 Markdown 继续作为用户可读、可编辑、可锁定的业务记忆载体。
8. 每次自动记忆变化都有任务轨迹可追溯。
9. 原 JSON ReMem 功能保持兼容。
10. 全量测试通过。

## 20. 最终产品定义

重构完成后，pm-mem 的产品定义为：

```text
pm-mem 是一个基于 ReMem 方法论的领域无关自演化记忆引擎，
支持角色注入、存储后端注入和可审计记忆演化轨迹。

短剧创作系统是 pm-mem 的业务实现之一，
它通过编剧、制片人、分镜师、一致性校验员等角色，
共同维护故事、人物、剧情、剧本和分镜等分层记忆。
```

该方案的核心价值在于：

- ReMemAgent 负责“如何演化记忆”。
- MemoryRole 负责“以谁的专业视角演化记忆”。
- MemoryStore 负责“记忆落在哪里以及如何被保护”。
- 业务工作流只负责“发起什么任务以及带什么上下文”。
